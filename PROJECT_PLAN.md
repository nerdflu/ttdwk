# Things To Do With Kids - Project Plan

## Current State

The repository is a static site built with Jekyll (v3.9.5) and hosted on GitHub Pages via a custom GitHub Actions workflow.

**Data Architecture:**
- **Collections:**
  - `_cities/`: Markdown files representing locations (e.g., `sydney.md`).
  - `_activities/`: Markdown files for specific attractions (e.g., `taronga-zoo.md`).
- **Data Files:** `_data/categories.yml` acts as the source of truth for the taxonomy.
- **Category Pages:** Dynamically generated via a Python script (`scripts/gen_city_category_pages.py`) which processes combinations of cities and categories, outputting to `category_pages/`.
- **CSS:** Vanilla CSS located in `assets/styles.css`.

## Problems Found

1. **Broken Local Build (Encoding Issue):** The default Jekyll environment assumes `US-ASCII` encoding on macOS. Because no explicit `theme` is set in `_config.yml`, GitHub Pages implicitly loads `jekyll-theme-primer`, which contains UTF-8 characters in its SCSS, crashing the build.
2. **Generated Files Tracked in Git:** The `category_pages/` directory is currently tracked in source control, even though it's re-generated by the Python script in the GitHub Actions pipeline. This leads to noisy commits and potential merge conflicts.
3. **Duplicate/Dead Code:**
   - Dead CSS: `.pill-list` classes in `assets/styles.css` are completely unused by the current HTML layouts (which now use `.cat-list`).
   - Duplicate Assets: Both `sydney.jpg` and `sydney.png` exist in `assets/cities/`, but only the `.jpg` is used by the `sydney.md` frontmatter.
4. **Duplicate Meta Tags:** `_layouts/default.html` manually generates a `<title>` tag, but also includes `{% seo %}` which generates a second `<title>` tag, violating HTML standard practices.
5. **Local Dev Friction:** There is no local `.gitignore`, and developers have to manually remember to run the Python script before starting the Jekyll server, otherwise the local category pages can become stale.

## Recommendations (Keep, Remove, Rename)

- **Keep:** The Python script for generating category pages. While a custom Ruby generator plugin is the "Jekyll way", since this is a solo project and the GitHub Actions workflow natively runs the Python script seamlessly, it remains simple and transparent. Keep the dual `_cities`/`_activities` collection structure.
- **Remove:** Untrack the `category_pages/` directory from Git and add it to a new `.gitignore` file. Remove the implicit Jekyll theme to speed up builds and fix the local encoding crash. Remove the unused `.pill-list` CSS and `sydney.png` image.
- **Rename:** No major renaming is needed at this stage; the naming conventions (`city_title`, `city_slug`) are clear and functional for the current architecture.

## Proposed Architecture & Implementation Phases

The goal is to stabilize the existing architecture to be simple, fast, and robust for a solo maintainer, without introducing heavy new frameworks.

### Phase 1: Clean Up & Stabilize
- **Fix Config:** Add `theme: ""` (or completely clear it) to `_config.yml` to prevent the implicit Primer theme from loading.
- **Gitignore:** Add a `.gitignore` file to ignore `_site/`, `vendor/`, `.bundle/`, and `category_pages/`. Run `git rm -r --cached category_pages/` to remove the currently tracked pages.
- **Prune Dead Code:** Delete `sydney.png` and remove `.pill-list` from `styles.css`.
- **Fix SEO Tags:** Refactor `_layouts/default.html` to pass the `_computed_title` into the Jekyll SEO tag rather than duplicating the `<title>` element.

### Phase 2: Local Developer Experience
- **Dev Script:** Create a simple shell script (e.g., `bin/dev` or a `Makefile`) that automatically runs the Python category generator and then starts `bundle exec jekyll serve --livereload`.

### Phase 3: Future Expansion (Events)
- **New Collection:** When ready for events, introduce an `_events/` collection structured identically to `_activities/` but with `start_date` and `end_date` frontmatter.
- **Taxonomy Integration:** Update the Python script to optionally pull events into the city category pages, or build a dedicated event generator script.
